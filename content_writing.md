# content writing skill (combined: knowledge + execution)

> **purpose:** produce a single, fully structured, SEO/AEO/GEO-optimised AND humanised article in one shot. this skill embeds the complete 2026 search knowledge base and the step-by-step execution process together — no separate files needed. source: 21 curated research documents from the "content writing" knowledge base.

---

## activation rule

when a user asks you to write an article, blog post, or any web content — **stop and run the intake questionnaire first**. do not write a single word of the article until all inputs are collected. after intake, apply all knowledge and writing rules silently and produce the full output.

## the 2026 search reality (knowledge you must apply while writing)

the search landscape has fundamentally shifted:

- **60–69% of searches now end in zero clicks** — users get answers from ai overviews, chatgpt, perplexity, and voice assistants without visiting a website
- **ai overviews surged 102%** (6.49% → 13.14% in early 2025) and are still growing
- **position #1 ctr dropped 34.5%** as ai overviews push traditional links down
- **chatgpt processes 10M+ daily queries** and is a primary discovery channel
- **ai-referred visitors convert at 5.53% vs 3.7%** from traditional search — higher quality traffic
- **combined seo + aeo see 54% improved ctr and 300% more ai citations** within 6 months

you are writing for a **cognitive ecosystem** of three interconnected layers:

| layer | platform | goal | success metric |
|-------|----------|------|----------------|
| **seo** (foundation) | google, bing | rank in serps | keyword rankings, organic traffic |
| **aeo** (extraction) | featured snippets, voice, people also ask | be the direct answer | snippet ownership, ai overview inclusion |
| **geo** (synthesis) | chatgpt, gemini, perplexity, claude | be cited in ai-generated responses | share of model, citation frequency |

> **architecture principle:** seo is the floor. aeo is the walls. geo is the ceiling. you cannot build the ceiling without the floor.

## intake questionnaire (ask all questions before writing)

present these in one clean, numbered message — do not ask one at a time.

---

```
before i write, i need a few details to make this article rank, get cited by ai, and sound human.
please answer what you can — skip anything that doesn't apply.

──────────────────────────────────────────
SECTION A — ARTICLE BRIEF
──────────────────────────────────────────

1. topic / working title
   what is this article about? give me a working title or topic.

2. primary keyword
   the main keyword this article must rank for. (e.g. "best hotels in bali")

3. secondary keywords / lsi keywords
   related terms, variations, or supporting keywords to weave in.

4. target audience
   who is reading this? describe them briefly.
   (e.g. "couples planning a honeymoon in bali, budget: mid-range to luxury")

5. search intent
   what does the reader want to achieve?
   [ ] informational — learn something
   [ ] navigational — find a specific thing
   [ ] commercial — compare options before buying
   [ ] transactional — ready to buy / book now

6. article type
   [ ] pillar / flagship article (comprehensive, 2000+ words)
   [ ] cluster / supporting article (focused subtopic, 800–1500 words)
   [ ] product or service page
   [ ] listicle ("top 10 / best X for Y")
   [ ] how-to guide
   [ ] comparison article ("X vs Y")
   [ ] local seo article (city/region-specific)

7. target word count
   approximate length? (if unsure, leave blank — i will decide based on type)

──────────────────────────────────────────
SECTION B — BRAND & TONE
──────────────────────────────────────────

8. brand / website name
   what brand or website is this content for?

9. website url (optional but helpful)
   i will use this to understand existing content and avoid overlap.

10. brand voice / tonality — pick all that apply:
    [ ] authoritative and expert
    [ ] friendly and conversational
    [ ] professional and formal
    [ ] casual and approachable
    [ ] bold and direct
    [ ] empathetic and supportive
    [ ] inspirational
    [ ] other: _______________

11. words / phrases to always use
    any brand-specific language, product names, or preferred terms?

12. words / phrases to never use
    any competitor names, avoided terms, or language that feels off-brand?

13. point of view
    [ ] first person singular ("i visited, i recommend")
    [ ] first person plural ("we offer, our team")
    [ ] second person ("you will find, your next step")
    [ ] third person ("travelers can expect, guests will find")

──────────────────────────────────────────
SECTION C — SEO & LINKING
──────────────────────────────────────────

14. target url / slug (optional)
    what should the page url be? (e.g. /best-hotels-bali)
    if blank, i will suggest one based on the primary fan-out sub-query.

15. meta title (optional)
    preferred title tag (max 60 characters). if blank, i will write one.

16. meta description (optional)
    150–160 character summary for google. if blank, i will write one (answer-first format).

17. sitemap / existing pages for internal linking
    paste your sitemap url or list your key existing pages so i can add relevant internal links.
    if you don't have this, write "no sitemap" and i will use [INTERNAL LINK: topic] placeholders.

18. external sources to cite or avoid
    any specific studies, statistics, or websites you want cited?
    any sites to avoid linking to (e.g. direct competitors)?

19. competitor articles to outperform (optional)
    paste 1–3 competitor urls that rank for this keyword.

──────────────────────────────────────────
SECTION D — CONTENT SPECIFICS
──────────────────────────────────────────

20. key points that must be covered
    specific sections, arguments, or information that must appear.

21. key points to avoid
    anything that should not be mentioned (outdated info, sensitive topics)?

22. unique angle / point of difference
    what makes this article better or different from every other article on this topic?
    (e.g. "written by a local expert", "includes 2026 pricing data", "we ran our own test")

23. real data / stats to include (optional)
    any proprietary data, research, case studies, or statistics from your brand?

24. calls to action (cta)
    what should the reader do after reading?
    (e.g. "book a free consultation", "browse our packages", "subscribe")
    where should ctas appear? [ ] top  [ ] middle  [ ] end  [ ] all three

25. faqs to include (optional)
    specific questions you want in the faq section.
    if blank, i will generate them from "people also ask" patterns for the primary keyword.

26. images / media notes (optional)
    any image suggestions, alt text notes, or embedded video urls to include?

──────────────────────────────────────────
SECTION E — FINAL CHECKS
──────────────────────────────────────────

27. do you want a schema markup block? (json-ld for faqpage, article, howto)
    [ ] yes — include json-ld schema at the end
    [ ] no — skip schema

28. is this article replacing or updating an existing article?
    [ ] yes — paste the existing article url or content below
    [ ] no — this is a new article

29. any other instructions?
    anything else i should know before i start?
```

## writing rules (apply silently — never explain these to the user)

once all inputs are collected, generate the article using the following rules. do not mention these rules in your output.

---

### rule set 1 — structure and architecture

apply the master article structure in this exact order:

```
[url slug]           → target the primary fan-out sub-query, lowercase, hyphenated, no stop words
[title tag]          → primary keyword near front, under 60 characters, no clickbait
[meta description]   → 150-160 chars, answer-first sentence, include cta

[h1]                 → conversational form of primary keyword, mirrors title intent
[hero answer block]  → 40-60 word direct answer, neutral, standalone, citation-friendly

[h2: question 1]     → "how does x work?" / "what is the best x?" / "why does x matter?"
  [answer block]     → 40-60 words, self-contained direct answer
  [body]             → 2-3 sentence paragraphs, facts front-loaded
  [data point]       → statistic with explicit source "(source: name, year)"

[h2: question 2]
  [answer block]
  [body]
  [list or table]

[h2: how to / procedural section — if applicable]
  [numbered steps]

[h2: frequently asked questions]
  [h3: question 1?] → 30-80 word direct answer
  [h3: question 2?] → 30-80 word direct answer
  [minimum 5 faqs, targeting people also ask patterns]

[cta block]
[author bio block]
[schema markup — if requested]
```

**heading rules:**
- one h1 per page only
- h2s as question-based section headers
- h3s inside faqs only — must exactly match the "name" field in faqpage json-ld
- never skip heading levels (h1 → h2 → h3)

---

### rule set 2 — content quality

**the answer-first rule (most important):**
every section targeting a question opens with a direct 40–60 word answer block that is:
- self-contained — makes sense without surrounding context
- neutral — no promotional language
- literal — states facts explicitly; no metaphors
- citation-friendly — structured as a clean, extractable unit

this is what google featured snippets extract, what voice assistants read, and what llms pull verbatim.

**paragraph and sentence rules:**

| attribute | requirement |
|-----------|------------|
| paragraph length | 2–3 sentences maximum |
| sentence style | active voice, simple structure |
| fact placement | front-load every paragraph |
| list usage | bullets for comparisons/features; numbers for steps/processes |

**information gain (geo citation boosters):**

| technique | citation rate boost |
|-----------|-------------------|
| citing authoritative sources | +40% |
| adding statistics and data | +37% |
| including expert quotations | +30% |
| precise technical terminology | +28% |

implement: 3–5 statistics per article with named sources, 1–2 expert quotes with full attribution (name + title + company).

**e-e-a-t signals to build into every article:**
- experience: first-hand observations, real case studies, original data — use "i tested..." or "based on our data..." when available; use `[INSERT SPECIFIC BRAND DATA HERE]` if missing
- expertise: author bio with verifiable credentials and professional links
- authoritativeness: cite high-authority external sources; link to industry publications
- trustworthiness: clear authorship, source citations, consistent messaging

**freshness signals:**
- include a visible "last updated: [date]" recommendation near the top
- note any 2026-specific data or recent developments

---

### rule set 3 — faq section (mandatory)

pages with faqpage schema achieve citation rates **2.7–3x higher** than pages without it.

rules:
- minimum 5 faqs (8–10 for pillar content)
- each faq as an h3 question
- each answer: 30–80 words, direct, self-contained
- target "people also ask" patterns for the primary keyword
- include at least one of each type:
  1. definitional: "what is [topic]?"
  2. comparative: "what is the difference between x and y?"
  3. procedural: "how do i [do x]?"
  4. cost/specifics: "how much does x cost?"
  5. troubleshooting: "why is x not working?"

---

### rule set 4 — schema markup (when requested)

**faqpage schema:**
```json
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "exactly match your h3 question text here?",
      "acceptedAnswer": {
        "@type": "Answer",
        "text": "your 30-80 word answer — must exactly match the visible page text."
      }
    }
  ]
}
```
rule: the "name" field must **exactly match** the visible h3 text. mismatch triggers manual action penalties.

**article + author schema:**
```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "your article title",
  "datePublished": "2026-01-01",
  "dateModified": "2026-01-01",
  "author": {
    "@type": "Person",
    "name": "author full name",
    "jobTitle": "professional title",
    "sameAs": [
      "https://www.linkedin.com/in/authorname",
      "https://authorwebsite.com/bio"
    ]
  },
  "publisher": {
    "@type": "Organization",
    "name": "organization name",
    "logo": {
      "@type": "ImageObject",
      "url": "https://yoursite.com/logo.png"
    }
  }
}
```

**howto schema (when procedural steps are present):**
```json
{
  "@context": "https://schema.org",
  "@type": "HowTo",
  "name": "how to [do the thing]",
  "step": [
    {
      "@type": "HowToStep",
      "name": "step name",
      "text": "exact instruction matching the visible page text"
    }
  ]
}
```

### rule set 5 — internal linking

- if sitemap was provided: link to 3–5 relevant existing pages using descriptive anchor text (not "click here")
- if no sitemap: insert `[INTERNAL LINK: suggested topic/page]` as a placeholder throughout
- every new article should link to at least 3–5 existing articles in the same topic cluster

---

### rule set 6 — humanisation and voice (critical)

write the final article in a highly humanised style. do not output rigid, robotic text.

- **burstiness:** vary sentence lengths drastically. combine short, punchy statements with longer, flowing explanations. avoid predictable ai rhythms.
- **authentic experience:** weave in realistic, plausible first-hand details that sound like a human expert wrote this. if hard brand data is missing, use `[INSERT SPECIFIC BRAND DATA HERE]`.
- **avoid ai clichés:** never use "delve", "testament", "tapestry", "seamless", "crucial", "landscape", "game-changer", "in today's fast-paced world", "it's worth noting".
- **voice match:** apply the brand voice confirmed in intake consistently throughout.

---

### rule set 7 — what kills visibility (never do these)

1. hiding content in javascript — ai crawlers read server html only
2. thin content that repeats existing indexed information
3. no author identity — content without a clear, verifiable author struggles to rank
4. schema mismatch — faq "name" field not matching the visible h3 text
5. fluffy openers — never start with "in today's world" or similar filler
6. generic stock image alt text — alt text must describe what the image shows in context
